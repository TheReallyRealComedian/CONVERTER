# UI-Inventur: markdown_converter (2026-05-10)

**Methodik:** Stufe 1 der Duan-Kaskade (analog F-1.1 / F-2.1 / F-3.1 / F-4.1). **Code-only** Inventur — kein Live-Walkthrough in diesem Sub-Thread (Container läuft auf Mintbox, aber für die Schwester-Feature-Inventur reicht statisches Reading + F-1.1-Vorlage; visuell-laufzeitabhängige States sind als „Code-deduced, nicht live verifiziert" markiert). `markdown_converter` ist **Schwester-Feature zu `document_converter`** (F-1-Welle), daher die zusätzliche **F-1-Korrespondenz-Spalte** in der Element-Tabelle und eine Übersicht am Doc-Ende, welche der 14 F-1-Patterns hier direkt / teil / nicht anwendbar sind.

**Feature-Kontext (aus CLAUDE.md):** Single-User-App, LAN-only, login-protected. Primärer `markdown_converter`-Workflow: Markdown im Textarea schreiben oder per File hochladen → Live-Preview (Client-side `markdown-it` im Iframe) → optional Style-Theme + Page-Orientation wählen → "Convert to PDF" submit → Browser-Download. Sekundärer Pfad: "Save to Library" speichert den **Markdown-Quelltext** (nicht das PDF) als `markdown_input`-Conversion. Drittes Sub-Feature: Reader-Mode mit eigener Toolbar (Font-Size, Width, Dark Toggle), Prefs in `localStorage`.

**Quellen:** [templates/markdown_converter.html](templates/markdown_converter.html), [static/js/markdown_converter.js](static/js/markdown_converter.js), [app_pkg/markdown.py](app_pkg/markdown.py), [templates/base.html](templates/base.html), [static/js/_utils.js](static/js/_utils.js), [static/css/style.css](static/css/style.css) (für State-Klassen `.c-btn`, `.c-btn--primary`, `.c-input`, `.c-alert`, `.c-alert--danger/success/warning/info`, `.save-library-btn`, `.save-library-btn.saved`, `.reader-mode-btn`, `.reader-toolbar`, `.preview-iframe`, `.markdown-textarea-wrapper`, `.orientation-btn` (inline im Template, nicht in style.css), `.toast-notification`, `.hidden`).

## Markdown-spezifische States

States, die in F-1 keinen Korrespondenten haben (durch das fehlende Iframe-Live-Preview, fehlende Style-Wahl, fehlenden Reader-Mode):

- **Iframe-Render-Lifecycle:** `srcdoc`-Update-Sequenz pro Keystroke / pro Theme-Change / pro Reader-Mode-Toggle. States: (a) initial empty (sample default), (b) typing (re-render auf jedes `input`-Event, kein Debounce), (c) theme-loading (CSS-Fetch läuft, Iframe re-rendert mit altem Theme weiter), (d) markdown-render-error (Exception in `md.render()` → Inline-Body-Error in Iframe), (e) library-load-error (`markdownit === undefined` → Inline-Body-Error in Iframe), (f) reader-mode-on (Font-Size + Width werden in `buildIframeDoc()` zusätzlich gesetzt).
- **Code-Highlighting-Asymmetrie:** Server-Pipeline (PDF) nutzt Pygments mit `noclasses=True` (inline-styles im `<pre>`), Client-Preview nutzt nackten markdown-it ohne Highlight-Plugin → Code-Blöcke sehen in Preview vs. PDF unterschiedlich aus.
- **HTML-Sanitization-Asymmetrie:** Server-Pipeline (PDF) sanitized mit `nh3.clean(...)`, Client-Preview rendert markdown-it mit `html: true` ohne Sanitization → HTML im Markdown wird in der Preview gezeigt, im PDF nur die in der Allow-List enthaltenen Tags.
- **Wide-Table-Detection:** Server-Pipeline ergänzt `<div class="landscape-table">`-Wrapper für Tabellen mit ≥6 Spalten **+** zur Laufzeit per JS in der Playwright-Page (Zeile 210-220), Client-Preview wraps nicht.
- **CSRF-Token-Lifecycle:** Submit-Handler refresht Token via `/api/csrf-token`-Endpoint vor dem echten `form.submit()` — sonst würde ein länger geöffneter Tab mit altem Token 400 sein.
- **Flash-Roundtrip:** PDF-Generation-Errors gehen über Server-Flash + Redirect (Empty-Content, Invalid-Filename, Style-Not-Found, PDF-Gen-Crash) → Browser navigiert auf neue Seite → Banner erscheint nach Redirect. **Im Gegensatz zu F-1, wo Errors per fetch in-page bleiben.**
- **Reader-Mode-Pref-Persistenz:** `localStorage.readerPrefs` hält `dark`, `fontSize`, `width` — aber **nicht** den On/Off-State des Reader-Mode selbst.

## Code↔live-Divergenzen-Verdacht (↯, Code-deduced)

(Konkrete Stellen wo Template, JS, und Route-Handler nicht zusammenpassen oder wo Code-Behavior live anders aussehen wird:)

1. **Initial-Sample-Markdown wird mit Server-`markdown_text` zusammengeführt** statt ersetzt. Template hat sample-Text **vor** `{{ markdown_text or '' }}`-Punkt (Zeile 43-69), kein `markdown_text`-Conditional am Anfang. Im Error-Pfad der `convert_markdown`-Route (Zeile 246) wird `render_template('markdown_converter.html', markdown_text=markdown_text)` ohne `themes`/`accepted_extensions`-Context aufgerufen → Sample-Text wird beim Re-Render plus User-Inhalt sichtbar (siehe Befund 4).

2. **PDF-Generation-Error-Re-Render der Page hat unvollständigen Context.** `render_template('markdown_converter.html', markdown_text=markdown_text)` (markdown.py:246) übergibt **nur** `markdown_text`. Die anderen drei Context-Vars (`themes`, `accepted_extensions`, `accepted_extensions_accept`) sind in der Template-Signatur erforderlich (`{% for theme in themes %}`-Loop, `accept="{{ accepted_extensions_accept }}"`, `{{ accepted_extensions|tojson }}`). **Bei Flask's Default-`Undefined` würde `{% for ... in undefined %}` einen `UndefinedError` werfen** → secondary 500. **Akut-flag-Kandidat** (siehe Befund 16).

3. **`updateStyle()` wird zweimal beim Page-Load aufgerufen** (markdown_converter.js:281 + markdown_converter.js:289) → zwei initiale Theme-CSS-Fetches (siehe Befund 6).

4. **Submit-Button-Loading-Text wird vor `form.submit()` zurückgesetzt** (markdown_converter.js:345-347) → User sieht "Preparing…" nur kurz während des CSRF-Refresh-Roundtrips, dann sofort zurück zu "Convert to PDF" — die Server-PDF-Gen-Zeit (mehrere Sekunden für Playwright-Browser-Boot) hat **keinen Loading-Indikator** sichtbar (siehe Befund 12).

5. **Esc-Key-Listener ist Document-global**, auch wenn `<textarea>` den Fokus hat — Esc beendet Reader-Mode, falls aktiv. Bei Textarea-Fokus + Reader-Mode-aktiv könnte Esc User unerwartet aus dem Reader-Mode rauswerfen (siehe Befund 17).

6. **`window.open` für Notion-Pfade** existiert hier **nicht** (kein Notion-Integration auf dieser Page), daher kein F-3.1-Befund-14-Korrespondent.

## Live-Walkthrough-Lücken (states die nur runtime sichtbar wären)

- **Iframe-Flicker zwischen `srcdoc`-Updates:** ob bei jedem Keystroke ein sichtbares Flackern entsteht — kann je nach Browser variieren. Code-deduced.
- **Reader-Mode-Visual:** Layout-Wechsel auf `.main-container.reader-mode` mit ausgeblendeter `.editor-pane` und re-styled `.preview-pane` — nicht visuell verifiziert.

- **PDF-Download-Verhalten nach Submit:** ob der Browser nach `form.submit()` mit `Content-Disposition: attachment` die Page **wirklich** stehen lässt (kein Reload, kein Navigate-Away) und nur den Download startet — Browser-spezifisch. Code-deduced.
- **Browser-Native-Validation-Popover für leeres `output_filename`-Feld:** Anker existiert (sichtbarer Input), aber visuelle Position/Style ist nicht aus Code ableitbar.
- **Theme-CSS-Fetch-Latenz:** Wie lang die Iframe ohne Theme-CSS rendert, bevor das `currentThemeCSS` gesetzt ist — Server-Latenz-Frage.
- **Globaler Theme-Toggle (#1) vs. Reader-Dark (#36)-Interaktion:** Reihenfolge-abhängig, ob beim Wechsel von Reader→non-Reader das `data-theme="dark"` korrekt entfernt wird wenn der globale Theme dunkel ist (Code Zeile 43: `document.documentElement.removeAttribute('data-theme')`) — siehe Befund 11.
- **CSRF-Refresh + concurrent Browser-Tab-Reload:** Wenn die `/api/csrf-token`-Anfrage länger braucht als der User zum Re-Submit-Click, ob doppelte Submits möglich sind. Code-deduced: `dataset.tokenRefreshed`-Flag verhindert das nach dem ersten Listener-Aufruf, aber innerhalb desselben Listener-Aufrufs ist der Race möglich.
