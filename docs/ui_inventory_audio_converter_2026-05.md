# UI-Inventur: audio_converter (2026-05-03)

**Methodik:** Stufe 1 der Duan-Kaskade. Statische Analyse + Live-Walkthrough.
**Live-Walkthrough:** teilweise erfolgreich. Container `markdown-converter-web` wurde via `docker compose up -d --build` frisch gestartet. Browser via `mcp__Claude_in_Chrome` (Linux-Chrome auf Mintbox), Tab nach Login auf `/audio-converter`. Tab-Switching, File-Tab Empty-Submit, File-Tab Conversion-Error (echte 500 von Deepgram bei 46-byte WAV), File-Tab Success+Save (über fetch-Mock simuliert, da Deepgram-Credit nicht verbrannt werden sollte), Podcast-Tab Empty-Submits und Custom-Prompt-Toggle, Drag-Drop-Versuch (synthetisches DragEvent), Live-Tab WebSocket-Auth-Failure (über fetch-Mock mit gefälschtem Token) verifiziert. **Nicht live verifiziert:** echter Mic-Permission-Granted-Pfad mit laufender Live-Transkription, echte Podcast-Generierung (würde Gemini-TTS-Credit verbrauchen), >500 MB-Hartcheck im Backend, Backend-Stop mid-transcription. Native OS-Drag-Drop einer echten Datei nicht testbar (das `mcp__Claude_in_Chrome__file_upload`-Tool wurde von Chrome mit "Not allowed" zurückgewiesen, wie schon bei F-1; statt dessen Datei via `DataTransfer`-API in den File-Input injiziert + `change`-Event dispatched).

**Quellen:** [templates/audio_converter.html](templates/audio_converter.html), [static/js/audio_converter.js](static/js/audio_converter.js), [app_pkg/audio.py](app_pkg/audio.py), [services/deepgram_service.py](services/deepgram_service.py), [templates/base.html](templates/base.html), [static/js/_utils.js](static/js/_utils.js), [static/css/style.css](static/css/style.css) (für State-Klassen `.tab-active/.tab-inactive`, `.lang-active/.lang-inactive`, `#mic-button`, `#mic-button.recording`, `.mode-radio-input`/`.mode-radio-label`, `.prompt-editor-content[.expanded]`, `.c-input`, `.c-btn`, `.c-drop-zone`, `.c-alert`, `.save-library-btn[.saved]`, `.toast-notification`).

## Bemerkungen / mögliche Bugs (separat — nicht in Tabelle gemischt)

Diese Auffälligkeiten gehen **über fehlende States hinaus** und sind echte Implementierungs-Befunde. Stage 2 muss entscheiden, welche davon Heuristik-Findings werden und welche separate Bug-Tickets:

1. **Drag-Drop ist nicht implementiert (Drop-Zone-Label lügt).** Die `<label class="c-drop-zone">` für File-Upload zeigt "Click to select an audio file or drag and drop", aber `audio_converter.js` enthält keine `dragover/dragleave/drop`-Handler. Synthetische DragEvents werden ignoriert. Real testen können wir nur einen OS-Drag, der vom MCP-Tool blockiert wurde — aber selbst dann würde das Browser-Default-Verhalten "Datei wird in der Tab geöffnet, nicht in den Input gelegt" greifen, weil der Input keine drop-Listener hat. **Inkonsistenz zu document_converter**, das echtes Drag-Drop hat.

2. **Empty-Submit auf File-Tab ist visuell silent.** Klick auf "Transcribe File" ohne Datei: HTML5-Validation-Message gesetzt ("Wähle eine Datei aus."), aber Anker = `display:none`. Im Walkthrough sah man **gar nichts** passieren. **Identisch zum document_converter F-1-Befund** (dort als Pattern 1 "Empty-Submit-Banner" in Cluster B gefixt; hier noch offen).

3. **`Save to Library` Stale-Visual-State (#30 + #21).** Selbe Mechanik wie document_converter F-1.B3-Bug: nach erfolgreichem Save bleibt `.saved`-Klasse hängen, nach neuer Transkription wird `disabled=false` und Text="Save to Library" gesetzt — aber **`.saved` wird nicht entfernt**. Resultat: grün-getönter Button mit Text "Save to Library". (Code-Stellen: [static/js/audio_converter.js:589-590](static/js/audio_converter.js:589) save-trans und [:626-628](static/js/audio_converter.js:626) save-live.)

4. **Drei konkurrierende Error-UX-Patterns auf einer Seite:**
   - Browser-native `alert()` für: WS/Token-Failure, Mic-Permission-implicit-Failures, Empty-Submit-Validierungen, Copy-Failures, Save-Failures, Script-Generation-Failures, Podcast-Generation-Failures (siehe #46) — **11+ Call-Sites**.
   - Inline `<span style="color: var(--nm-danger)">…</span>` direkt in `#transcription-result-container` für Conversion-Failure (#25/#27).
   - Server-side `c-alert--danger` Banner für Configuration-Error (#11).
   Ohne Verwendung des `showAlert()`-Helpers aus `_utils.js`, der für document_converter in F-1 Cluster B/C eingeführt wurde. Auch kein `showToast()` für Save-Success.

5. **Mic-Button hat keinen Loading-State.** Click → Token-Fetch → WS-Open → getUserMedia. Drei serielle async-Steps, optisch komplett silent. Bei langsamer Netzwerk-Verbindung wirkt das wie "Button reagiert nicht". Vergleich: document_converter F-1 Cluster Polish-2 hat explizit Loading-State auf Drop-Zone eingeführt — Mic-Button braucht das auch.

6. **Mic-Button hat keinen sichtbaren `:focus-visible`-Style.** Nur `:hover` und `.recording` sind im CSS definiert. Tab-Fokus auf den Mic-Button ist visuell ununterscheidbar von hover/idle. Für die Hauptaktion der Live-Tab problematisch.

7. **Mode-Radios (Format Monolog/Dialogue) sind keyboard-unzugänglich.** Input ist `display:none`, Label hat `tabindex=-1`. Tab-Pfad überspringt das komplette Format-Toggle. Da auch `:focus-visible`-Style auf der Label-Card fehlt, gäbe es selbst bei `tabindex=0` keinen sichtbaren Fokus.

8. **Custom-Prompt-Toggle (Advanced-Section) ist keyboard-unzugänglich.** `<div>` mit `cursor: pointer` und onClick-Handler, aber `tabindex=-1`, keine `role="button"`, kein `aria-expanded`. Klassischer "Div-as-Button"-Anti-Pattern.

9. **Lang-Buttons Disabled-State während Recording ist visuell nicht sichtbar.** `btn.disabled=true` wird gesetzt (während WS aktiv), aber kein `.lang-btn:disabled` CSS-Rule, kein Opacity-Dim, kein cursor-Wechsel. User klickt und nichts passiert — selbe silent-fail-Pattern wie #2.

10. **Live-Transcript Textarea ist editierbar während Recording.** Kein `readonly`, kein `disabled`. User könnte mitten im Stream tippen, was den `baseText`-Buffer im `socket.onmessage`-Append-Pfad korrumpiert (`transcriptOutput.value = baseText + transcript;` überschreibt User-Edits in jedem interim-Frame).

11. **Live-Tab "Clear" hat keine Confirmation, kein Undo.** Klick → `value=''` + `baseText=''` ohne Rückfrage. Bei langer Live-Transkription ist das ein 1-Klick-Datenverlust. Auch "Reset to Default" bei Custom-Prompt-Editor (#41) hat keine Confirm.

12. **Save-Live-Title nutzt User-locale `toLocaleDateString()` ohne Format-Spec.** Resultiert in Locale-abhängigen Strings ("3.5.2026" vs "3/5/2026" vs "5/3/2026"). Auch Mix EN ("Live Transcription") + DE-Date je nach Browser-Locale.

13. **File-Input `accept="audio/*"` filtert nur den Picker, nicht den Drag-Drop** (und Drag-Drop existiert eh nicht — siehe #1). Backend (#25-Pfad) macht keine Mime-Type-/Extension-Validation — leitet alles direkt an Deepgram. Die Validation passiert dann erst in Deepgram (silent 500 für non-audio). Ähnlich der F-1 Cluster-D Befunde, hier offen.

14. **Audio-Element (#44) hat keine Custom Loading/Error-States** — Browser-default reicht für playback, aber wenn die Blob-URL revoked wird (z.B. neue Generation), zeigt Audio-Element keinen Error-Hinweis.

15. **Podcast-Polling hat keinen Stop/Cancel-Knopf.** Loop läuft `while(status === 'pending' || 'processing')` mit 2s-Tick und nur `if (status === 'failed') throw` als Exit. Bei sehr langer Generation und User-Abbruchwunsch: nur Tab-Reload möglich. Counter wächst unbeschränkt ("Generating... (240s)" und weiter).

16. **Podcast Generate-Script Button-Restore innerHTML hat hartkodiertes Markup.** Im `finally`-Block wird `generateScriptBtn.innerHTML = '<svg…>Generate Script from Text Above'` mit dem gesamten SVG inline gesetzt — Code-Duplikation zur Template-Definition. Wenn das Template-Markup geändert wird, bleibt der Restore stale.

17. **Podcast Mode-Radio "selected"-Visualisierung ist sehr subtil.** Inset-Shadow-Wechsel + leichte Blau-Tönung (`var(--nm-tint-accent)`). In Screenshots schwer zu unterscheiden. User hat möglicherweise Schwierigkeit zu erkennen, welcher Modus aktiv ist (besonders auf älteren Displays).

18. **Configuration-Error-Banner (#11) ist generisch.** "Audio services are not configured. Please set API keys." nennt **nicht** welcher Key fehlt (nur Deepgram wird tatsächlich geprüft via `deepgram_api_key_set`). Auch wird der Gesamte Tab-Container hidden, was disproportional ist — Podcast-Tab funktioniert ohne Deepgram, Live/File brauchen ihn.

19. **Result-Container Copy-Bug (#27/#28).** `Nothing to copy!` Guard greift nicht beim Placeholder-Text "The transcription will appear here after processing." — User klickt Copy auf leerem Result und kriegt diese englische Phrase in die Zwischenablage. (Vergleichbar mit document_converter, wo der Result-Container hidden bleibt.)

20. **Englische Strings überall.** Sämtliche Microcopy auf der Seite (Tabs, Labels, Buttons, Placeholders, Alerts) ist EN. Inkonsistent zur DE-Lokalisierung in document_converter (F-1 Cluster Polish-1 hat dort 7 Strings übersetzt). Stage 2/3 muss entscheiden, ob hier flächendeckende DE-Übersetzung angesetzt wird.
