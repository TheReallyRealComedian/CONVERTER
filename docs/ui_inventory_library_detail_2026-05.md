**Methodik:** Stufe 1 der Duan-Kaskade (analog F-1.1 / F-2.1). **Code-only** Inventur — kein Live-Walkthrough verfügbar in diesem Sub-Thread (Master macht Walkthrough-Nachreichung in F-3.2 oder zwischen den Sprints, falls notwendig). ## Zusammenfassung

- **Gesamtzahl interaktive Elemente:** 21 (#1–#11, #13–#17, #20, #22–#24, #26, #29). Die Read-only-Sektionen #27/#28, dekorative #12/#21, Container #18/#19, dynamisch gerenderter Felder-Container #25 (intern interaktiv, aber als Container ein Wrapper), sowie Feedback-Mechanismen #30/#31/#32 sind System-Reaktionen / dekorativ / read-only und keine User-Interaction-Targets, werden aber zur Vollständigkeit gelistet (Gesamt-Tabellenzeilen: 32).

- **Im Code identifizierte fehlende States (✗):**
  - **Title-Input (#11) und Tags-Input (#29) haben keinen Loading- und keinen Error-State** — `updateField()` zeigt nur den Success-Toast und ignoriert `r.ok===false` und Network-Failures komplett (kein `.catch`, kein `if (!r.ok)`-Branch). Silent fail bei 4xx/5xx.
  - **Favorite-Button (#13) hat keinen Loading- und keinen Error-State** — `toggleFavorite()` toggelt nur bei `r.ok`, fängt aber Network-Failures gar nicht und zeigt keine Bestätigung.
  - **Favorite-Button (#13) hat kein eigenes `:focus-visible`-Style** — Browser-Default-Outline kommt zur Anwendung, aber kein neomorphism-Ring wie bei `.c-btn`.
  - **Download-Btn (#15) hat keinen Loading-, Error- oder Success-State** — keine Toast-Bestätigung, kein try/catch (identisch zum F-1-Befund auf document_converter).
  - **Open-in-Editor (#16) hat keinen Error-State** für `localStorage.setItem`-Failures (Quota, Disabled, Private-Mode).
  - **Delete-Btn (#17) hat keinen Loading-State** während des DELETE-Roundtrips und keinen Error-State bei DELETE-Fail (silent — wenn der Server 5xx zurückgibt, wird einfach **nicht** zur Library navigiert; User klickt wieder, ohne zu wissen, was passiert ist).
  - **Notion-Panel-Toggle-Button (#20) hat kein `aria-expanded`, kein `aria-controls`** — siehe Befund 6.
  - **Notion-Submit-Btn (#26) hat keine Pre-Validation** für Required-Fields (`title` ist als `required:true` markiert in den `fieldDefs`, aber das wird nirgends im Code geprüft — auch kein `required`-HTML-Attribut wird auf das `<input>` gesetzt).
  - **Notion-Submit-Btn (#26) hat keinen persistenten Error-Banner** — alle Fehler gehen über `showToast` (auto-dismiss 2.5s) statt über `showAlert(... 'danger', ...)`. Inkonsistent zu F-1 Cluster B/C, das genau dafür `showAlert` etabliert hat.
  - **Notion-Suggestions-Container (#25) hat keinen Loading-State** während `loadSuggestions()` läuft — datalists fehlen kurz, ohne User-Sichtbarkeit.
  - **Content-`<pre>` (#19) hat keine a11y-Annotations** (`tabindex`, `role`, `aria-label`) — Screenreader können den Output nicht gezielt fokussieren. Identisches Pattern wie F-1 / F-2.
  - **Tags-Input (#29) rendert keine Tag-Chips** — bleibt Plain-CSV. CSS hat zwar `.c-tag { background: var(--nm-tint-accent); }` definiert, wird aber im Detail-Template nicht verwendet.
  - **Empty-State `conversion.content == ''`**: das `<pre>` rendert leer, kein „Kein Inhalt"-Placeholder. (Praktisch nicht erreichbar, weil `api_create_conversion` `content` als required validiert, aber theoretisch möglich nach manuellem DB-Write.)

- **Code↔live-Divergenzen-Verdacht (↯, Code-deduced):**
  - **Sidebar-Active-State auf der Detail-Seite fehlt vermutlich** — `base.html` markiert nur `request.endpoint == 'library'` als aktiv, der Endpoint hier ist `library_detail`. Siehe Befund 1. **Master-Verifikation per Walkthrough nötig**, weil `base.html` nicht direkt eingelesen wurde (nur die Block-Strukturen).
  - **Notion-Panel Re-Toggle löscht User-Inputs** — `toggleNotionPanel()` ruft beim Open immer `selectTarget(DEFAULT_TARGET)`, was `renderNotionFields()` triggert, was `container.innerHTML = …` zuweist. User-Eingaben in den Feldern gehen verloren, wenn der Panel zwischendurch zugeklappt wird. Befund 5.
  - **Target-Switch löscht User-Inputs** — selber Mechanismus über `selectTarget()` → `renderNotionFields()`. Befund 7.
  - **Datum-Default in Notion-Felder ist UTC ohne Timezone-Anpassung** — `new Date().toISOString().slice(0, 16)` liefert `2026-05-09T13:42` als UTC, aber das `<input type="datetime-local">` erwartet **lokale** Zeit. User in Europe/Berlin sieht eine 1–2h zurück verschobene Zeit. Befund 15.
  - **Notion-Field-Textarea-Content nicht escaped** — `<textarea …>${f.value}</textarea>` interpoliert direkt; `f.value` kommt aus `#detail-title.value` (für `title`-Field) bzw. `#tags-input.value` (für `tags`-Field). Inhalte mit `</textarea>` würden den Block sprengen, mit `<` würden HTML-Render starten. **HTML-Injection-Risk** (Single-User-Kontext mildert, aber nicht harmlos). Befund 9.
  - **Toast-Level für Fehler ist „success"** — alle `showToast` Calls in `library_detail.js` lassen `opts` weg, damit ist `level='success'` (grün) — auch für `'Copy failed'`, `'Error: ...'`, `'Failed to connect to Notion'`. Visuell falsche Tönung. Befund 13.
  - **`window.open(url, '_blank')` ohne `noopener,noreferrer`** — Notion-URLs sind https://notion.so, also vermutlich unkritisch, aber das ist keine Verteidigung. Befund 14.

- **Unverifizierbare States (?):**
  - `:focus-visible`-Styles auf Sidebar-Links, Brand-Link, Theme-Toggle, Mobile-Sidebar-Toggle, Favorite-Btn, Notion-Toggle-Header, Notion-Submit-Btn — Code zeigt keine expliziten Regeln, Browser-Default-Outline wird zur Anwendung kommen.
  - Tatsächliche Sichtbarkeit der `.c-btn--primary`-Tönung gegenüber `.c-btn` für die Notion-Target-Buttons (#22–#24) — Inset-Wechsel ist im CSS subtil (`linear-gradient(145deg, #c8d0e8, #d8dde4)` vs. `var(--nm-bg)`), und beide haben `box-shadow: var(--nm-raised-sm)` — Master-Walkthrough sollte verifizieren ob der „aktiv"-State erkennbar ist.
  - Verhalten bei sehr langen Conversion-Inhalten (>100k chars) — `<pre>` mit `whitespace-pre-wrap` + `max-h-[calc(100vh-280px)]` + `overflow-auto` sollte scrollen, aber Browser-Performance bei Riesen-Strings nicht aus Code ableitbar.
  - Verhalten bei `confirm()` über Tastatur (Enter ↔ Cancel) — Browser-Default.
  - Notion-MCP-Server unreachable: `sendToNotion` Catch greift; Toast „Failed to connect to Notion". Aber: was passiert, wenn das Backend (`/api/conversions/<id>/send-to-notion`) den 502 schon ausspuckt? Dann läuft der Pfad in den 4xx-Branch und zeigt Toast „Error: Failed to reach Notion server." — ist also doppelt abgesichert, aber nur statisch verifiziert.

- **Unterschiede zu F-1.1 / F-2.1 in der State-Coverage:**
  - **Keine Drop-Zone, keine File-Inputs** — entfällt der ganze „silent-empty-submit"-Komplex.
  - **Keine Form-Submission im klassischen Sinn** — alles AJAX via `fetch`. Aber: `updateField` und `toggleFavorite` haben keine Loading/Error-States, was hier eine **Lücke ist, die in F-1/F-2 nicht so stark auffiel**, weil dort die Submission über einen sichtbaren Submit-Button mit „Converting..."-Text-Swap lief. Hier sind die Auto-Save-Pfade völlig unsichtbar.
  - **Keine konkurrierenden Error-UX-Patterns** wie in F-2 (Browser-`alert()` + Inline-Span + `c-alert`) — `library_detail` nutzt ausschließlich `showToast`. Inkonsistenz hier ist eine *andere*: Toasts werden auch für Fehler eingesetzt, mit falschem Level (s.o.).
  - **Keyboard-Zugänglichkeit** ist global solider (alle Buttons sind echte `<button>`-Elemente, keine `<div tabindex=-1>`-Anti-Patterns wie auf audio_converter), **aber** der „Save to Notion"-Header-Button hat kein `aria-expanded` — disclosure-pattern-spezifischer Mangel.

## Bemerkungen / mögliche Bugs (separat — nicht in Tabelle gemischt)

Diese Auffälligkeiten gehen **über fehlende States hinaus** und sind echte Implementierungs-Befunde aus dem Code-Reading. Stage 2 muss entscheiden, welche davon Heuristik-Findings werden und welche separate Bug-Tickets:

1. **Sidebar-Active-State fehlt auf Detail-Seite (Vermutung — Walkthrough-Verifikation nötig).** Die Sidebar-Nav in `base.html` markiert `Library` nur als aktiv, wenn `request.endpoint == 'library'`. Auf `library_detail` ist der Endpoint `library_detail`. Wenn die Active-Markierung tatsächlich nur auf exakten Endpoint-Match prüft, fehlt der visuelle „du bist im Library-Bereich"-Cue auf allen Detail-Seiten. (Code-Anker: nicht eingelesen — `templates/base.html` rund um die Nav-Links; Master macht Walkthrough-Verifikation in F-3.2.) Sev: niedrig (kosmetisch). Disposition: Finding + Bug-Ticket-Kandidat (Bug nur falls Walkthrough bestätigt).

2. **`updateField` und `toggleFavorite` haben kein Error-Handling.** Wenn `PUT /api/conversions/<id>` 4xx/5xx oder Network-Fail liefert, geht der Browser silent zurück in den idle-Look. User editiert Title, drückt Tab, bekommt **keinen** Toast, denkt „gespeichert". Tatsächlich nicht. Identische Mechanik beim Favorite-Toggle: Klick, kein Glyph-Wechsel, User klickt nochmal. (Code-Anker: [static/js/library_detail.js:8-18](static/js/library_detail.js#L8-L18) `updateField`; [static/js/library_detail.js:20-32](static/js/library_detail.js#L20-L32) `toggleFavorite`.) Sev: mittel (Datenverlust-Risiko silent). Disposition: Finding + Bug-Ticket.

3. **Delete-Pfad hat kein Error-Feedback.** `deleteConversion()` führt `DELETE` aus und navigiert nur bei `r.ok`. Bei 4xx/5xx oder Network-Fail bleibt der User auf der Detail-Seite — **ohne jeden Hinweis**, dass die Aktion fehlschlug. (Code-Anker: [static/js/library_detail.js:55-60](static/js/library_detail.js#L55-L60).) Sev: mittel. Disposition: Finding + Bug-Ticket.

4. **Toast-Strings sind alle EN.** `${field} updated` (variable, aber „updated" ist EN), `Copied to clipboard`, `Copy failed`, `Saved to Notion!`, `Error: <error>`, `Failed to connect to Notion`, `Sending...`, `Send to Notion`. Plus die Template-Strings (siehe unten). Insgesamt ~18 EN-Strings auf der Seite. Pre-Existing-Item, in Memory bekannt. Wird in F-3-IMPL-* nach Patterns-Sprint behoben — nur Erwähnung.

5. **Notion-Panel Re-Toggle wirft User-Inputs weg.** `toggleNotionPanel()` ruft beim Open immer `selectTarget(DEFAULT_TARGET)`, was die Felder neu rendert. Wenn der User Felder ausgefüllt hat, dann den Panel zugeklappt hat (z.B. um an den Title oben zu ändern), dann wieder geöffnet hat → leere Felder. (Code-Anker: [static/js/library_detail.js:66-75](static/js/library_detail.js#L66-L75).) Sev: mittel. Disposition: Finding + Bug-Ticket.

6. **Notion-Toggle-Header fehlt `aria-expanded` / `aria-controls`.** Klassisches Disclosure-Widget ohne ARIA-Annotation. (Code-Anker: [templates/library_detail.html:47-50](templates/library_detail.html#L47-L50).) Sev: niedrig (Single-User, aber a11y-Lücke). Disposition: nur Finding.

7. **Target-Switch löscht User-Inputs.** Identisch zu Befund 5 — `selectTarget()` ruft `renderNotionFields()`, das `container.innerHTML = …` setzt. User füllt Meeting-Felder aus, switcht aus Versehen auf Note-Target → alles weg. (Code-Anker: [static/js/library_detail.js:85-91](static/js/library_detail.js#L85-L91).) Sev: niedrig–mittel (User-Versehen-Klick eher selten). Disposition: Finding (Bug-Ticket eher in F-3-IMPL-Sammelpaket).

8. **`loadSuggestions` prüft nicht den HTTP-Status.** `fetch('/api/notion/suggestions').then(r => r.json())` — wenn das Backend 4xx/5xx mit JSON-Body liefert, wird der Error-Body als Suggestions interpretiert. Erst der `.catch()` greift bei Netzwerkfehler oder JSON-Parse-Fehler. Backend [app_pkg/integrations/notion.py:80-96](app_pkg/integrations/notion.py#L80-L96) gibt allerdings immer 200 mit Empty-Object zurück, also faktisch unkritisch — aber fragiler Code. (Code-Anker: [static/js/library_detail.js:77-83](static/js/library_detail.js#L77-L83).) Sev: niedrig. Disposition: nur Finding.

9. **Notion-Field-Textarea-Content nicht escaped (HTML-Injection-Risk).** In `renderNotionFields()` ist `<input value="${esc(f.value)}">` HTML-attribute-escaped, aber `<textarea …>${f.value}</textarea>` ist **roh interpoliert**. `f.value` kommt für `description`/`summary` aus leerem String (unkritisch), aber für andere Targets könnten künftige Erweiterungen User-Content reinleiten. Heute: `summary`-Field hat default `value: ''` — also faktisch nicht erreichbar. Beobachtung: der `esc`-Helper ist nur auf `value`-Attribute angewendet, nicht auf Textarea-Content. (Code-Anker: [static/js/library_detail.js:134-136](static/js/library_detail.js#L134-L136).) Sev: niedrig (heute unausnutzbar, aber Code-Hygiene). Disposition: nur Finding.

10. **Notion-Submit-Errors werden via `showToast` statt `showAlert` angezeigt.** `showToast` auto-dismissed nach 2.5s — wenn der Notion-Server 502 oder 4xx liefert mit User-relevanten Fehler-Details, hat der User nur 2.5s zum Lesen. F-1 Cluster B/C hat `showAlert` exakt für diesen Use-Case eingeführt (persistente Banner mit Close-Button). Inkonsistenz zur etablierten Pattern. (Code-Anker: [static/js/library_detail.js:171-175](static/js/library_detail.js#L171-L175).) Sev: niedrig. Disposition: Finding (Pattern-Konvergenz in F-3.3).

11. **File-Size-Anzeige in MB ohne KB/B-Fallback.** Template-Code `{{ "%.1f"|format(conversion.source_size_bytes / 1048576) }} MB` — gleiche Falle wie F-1 (Doc-Conv), aber nicht im Frontend lösbar via `formatFileSize()` (das ist JS-Helper, hier ist Server-Render). Sub-MB → „0.0 MB". (Code-Anker: [templates/library_detail.html:79](templates/library_detail.html#L79).) Sev: kosmetisch. Disposition: Finding (Cluster mit F-1.B-Pattern).

12. **Tags-Input rendert keine Chip-Visualisierung.** Plain CSV-Textfeld. CSS hat `.c-tag { background: var(--nm-tint-accent); }` definiert (Library-List-View nutzt es vermutlich). Detail-View lässt die Tags völlig unstyled als `<input>` stehen — kein visueller Hinweis was als Tag interpretiert wird, kein Add/Remove einzelner Tags. (Code-Anker: [templates/library_detail.html:112-115](templates/library_detail.html#L112-L115).) Sev: kosmetisch. Disposition: Finding (Pattern-Sprint F-3.3 ist die Stelle).

13. **Toast-Level falsch für Error-Pfade.** Alle `showToast`-Calls in `library_detail.js` lassen `opts` weg → default `level='success'` (grün). Auch für „Copy failed", „Error: ...", „Failed to connect to Notion". Visuell sieht der User einen grünen Toast mit „Error" — kognitiver Mismatch. (Code-Anker: [static/js/library_detail.js:36-38](static/js/library_detail.js#L36-L38), [:174](static/js/library_detail.js#L174), [:177](static/js/library_detail.js#L177).) Sev: mittel (UX-Verwirrung). Disposition: Finding + Bug-Ticket.

14. **`window.open(url, '_blank')` ohne `noopener,noreferrer`.** Standard-Praxis-Lücke. URL kommt aus Notion-API-Response (vertrauenswürdige Quelle), aber defensiv-defaulting ist Best-Practice. (Code-Anker: [static/js/library_detail.js:172](static/js/library_detail.js#L172).) Sev: niedrig (Notion-Domain trusted). Disposition: nur Finding.

15. **Datum-Default für Meeting-Notion-Field ist UTC, nicht lokal.** `new Date().toISOString().slice(0, 16)` liefert `2026-05-09T13:42Z` als UTC — das `<input type="datetime-local">` interpretiert das als lokale Zeit, also zeigt es 1–2h verschoben gegenüber „jetzt" für Europe/Berlin.
